  Line 141: Changed data[0x184] to data[0x186] for reported_tcb_snp.

  The AMD SEV-SNP spec defines the TCB_VERSION 8-byte structure as:

  ┌──────┬────────────────────┐
  │ Byte │       Field        │
  ├──────┼────────────────────┤
  │ 0    │ blSPL (bootloader) │
  ├──────┼────────────────────┤
  │ 1    │ teeSPL             │
  ├──────┼────────────────────┤
  │ 2-5  │ Reserved           │
  ├──────┼────────────────────┤
  │ 6    │ snpSPL             │
  ├──────┼────────────────────┤
  │ 7    │ ucodeSPL           │
  └──────┴────────────────────┘

  Our code was reading byte 4 (reserved, always 0) as snpSPL instead of byte 6.
  This caused the KDS URL to request a VCEK for snpSPL=0 when the real value was
   27, resulting in a VCEK whose public key didn't match the report's signing
  key.

  parsing.

  Line 121: Removed Accept: application/x-pem-file header from fetch_cert(). The

● The TDX stuff is in better shape than SNP was. Here's the breakdown:

  Custom crypto: minimal — signature verification delegates to p256 (ECDSA
  P-256) and rsa crates properly. No reimplemented crypto algorithms.

  Custom parsing: yes, but it's unavoidable — there's no equivalent of the sev
  crate for TDX quotes. The Intel DCAP quote format has to be parsed manually.
  The parsing uses the scroll crate for safe reads and has length checks.

  No cert chain validation — TDX quotes are self-contained (the attestation key
  and signature are inside the quote), so there's no AMD-style ARK→ASK→VCEK
  chain to validate.

  Fragility concerns (minor):
  - The quote parser has hardcoded offsets for header (48 bytes) + body (584
  bytes), but these match the Intel spec and have proper bounds checks
  - The tpm_common.rs TPM2B_PUBLIC and TPMS_ATTEST parsing (shared by az-snp and
   az-tdx) has some .unwrap() calls on slice conversions that could panic on
  malformed input, but these are after length checks
  - Auth data extraction in verify_quote_signature checks quote_bytes.len() < 
  body_end + 132 before accessing

  Bottom line: Nothing as bad as what SNP had (wrong byte offset causing silent
  verification failure). The TDX code is correct manual parsing of a
  well-defined binary format, not a reimplementation of an existing library. No
  action needed unless you want to harden the error handling on malformed
  inputs.

❯ what does the coco libs do for tdx, is there a tdx lib that they use?  

● Explore(Check CoCo TDX crate usage)
  ⎿  Done (25 tool uses · 64.0k tokens · 1m 0s)
  (ctrl+o to expand)

● The CoCo trustee uses intel-tee-quote-verification-rs from Intel's DCAP SDK
  for the actual signature verification:

  intel-tee-quote-verification-rs = {
      git = "https://github.com/intel/confidential-computing.tee.dcap",
      tag = "DCAP_1.24"
  }

  This is a Rust binding to Intel's C DCAP library — it calls tee_verify_quote()
   which handles:
  - ECDSA signature verification
  - Certificate chain validation (PCK cert chain)
  - TCB status evaluation
  - Collateral fetching (from Intel PCS)

  For quote parsing though, CoCo does the same thing we do — manual binary
  parsing with scroll::Pread on the QuoteHeader, ReportBody2, etc. There's no
  pure-Rust crate that parses TDX quotes into nice types like sev does for SNP.

  So the situation is:

  Column 1: Established crate
  SNP: sev (pure Rust, has everything)
  TDX: intel-tee-quote-verification-rs (C bindings to DCAP)
  ────────────────────────────────────────
  Column 1: What CoCo uses
  SNP: sev crate for parsing + verification
  TDX: DCAP lib for sig verification, manual parsing for fields
  ────────────────────────────────────────
  Column 1: What we do
  SNP: sev crate (after our refactor)
  TDX: Manual parsing + p256 crate for sig verification

  The main gap is that we do our own ECDSA P-256 signature verification on the
  quote instead of using Intel's DCAP library. Our approach works (uses p256
  crate correctly), but DCAP also validates the PCK cert chain and checks TCB
  status against Intel's collateral — things we skip.

  The tradeoff: intel-tee-quote-verification-rs is a C dependency (won't compile
   to WASM, harder to build), while our pure-Rust approach is portable. If you
  want the full DCAP validation, we'd need to add it as an optional dep.
  Otherwise what we have is reasonable — the quote parsing is correct and
  unavoidably manual since no pure-Rust TDX quote crate exists.
